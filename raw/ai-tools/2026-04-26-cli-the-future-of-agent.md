## CLI 最早要解决的：让人把命令准确交给机器

CLI（Command-Line Interface）不是"不够友好但更高效"的界面，而是为解决一个更基础的问题而生：人怎么把动作清楚地表达给机器。

终端和电传打字机出现后，第一次形成"输入一句，立刻看到结果"的实时回路。CLI 随之成形——把动作压成"命令词 + 参数 + 对象"。

三个一直没失效的能力：
1. **动作明确交出去**：把要做的动作压成一行命令
2. **反馈立刻拿回来**：实时看到执行结果
3. **多个动作继续接起来**：命令之间可以管道组合

## 三段历史证据

**rm**：说明 CLI 最基础的结构"命令 + 参数 + 对象"，从1970年代到现在没变过。

**Unix**：给了 CLI 一套生存哲学——文本进、文本出，每个命令只做一件事，命令之间可以接起来。这让 CLI 在 GUI 出现后没有被替代，而是在专业世界里继续承担连续动作。

**git**：GUI 时代来了以后，CLI 没有死。它在专业世界里一直承担连续动作和组合动作。

核心认知：**CLI 从一开始就是执行层**，不是前台界面。

## 为什么 Agent 时代又轮到 CLI

软件世界正在从"人操作软件"变成"人提出目标，Agent 拆任务、执行、修正"。

Agent 一旦真正开始干活，立刻碰到四个问题：
1. 命令怎么送进软件
2. 执行结果怎么拿回来
3. 出错以后怎么继续修正
4. 多个动作怎么接成一条链

**CLI 恰好是最成熟的答案**——不是因为它最新，而是因为在这四件事上积累最深。

## CLI 和 API 不是一回事

- **API**：软件能力的原始接口，是能力本身
- **CLI**：站在能力层上面的包装，把鉴权、参数拼装、分页、错误处理封装掉，对外暴露成一条条可执行的命令

CLI 的能力层可以是远程 API（如飞书 API），也可以是本地程序（如 rm 背后连的是操作系统）。

## 为什么 Agent 的执行层先收敛到 CLI

**第一，位置刚好。**
- GUI 对人友好，对 Agent 别扭——需要截图、识别界面、定位元素、模拟点击，成本高链条容易断
- 裸 API 太散——鉴权、分页、重试、错误码解析都得自己处理
- CLI 在中间：比 GUI 稳定（文本接口），比裸 API 省事（封装好了）

**第二，大模型天生会用。**
训练语料里充满了命令行指令：项目安装说明、构建脚本、技术问答、错误排查过程。模型不需要从零学习，对这种范式天然熟悉。函数调用常常是在一堆结构各异的接口之间切换，而命令调用是在统一的文本格式里做组合。

**第三，Unix 哲学契合 Agent 工作方式。**
- 文本是模型的母语
- stdin/stdout/stderr 通道稳定，exit code 表示成功或失败
- 每个命令只做一件事，多个命令可以接成一条链

```
cat access.log | grep ERROR | wc -l
```
这行命令本质上是"读日志→筛错误→做统计"三件事接成一条链。Agent 擅长的正是这种连续动作——拉数据、填数据、出图表，一步接一步。

## MCP 和 CLI 的关系

### 整体位置

两者都站在 Agent 和底层能力之间，但方式不同。

**CLI 路径**：
- Agent 必须先知道有什么命令（如 `lark-cli`）
- Agent 生成一行字符串命令
- 执行后以文本（通常是 JSON）吐回结果
- Agent 读文本决定下一步
- **前提**：Agent 不知道命令存在就走不通

**MCP 路径**：
- MCP Server 主动把能力清单推给 Agent
- Agent 看到清单，选工具、填参数、发起调用
- 不需要知道底下跑的是 `lark-cli` 还是直接 API
- **前提**：Agent 可以不知道底层细节

### CLI 的两个强项

**管道组合**：一次调用、三步操作通过管道接在一起
```bash
cat access.log | grep ERROR | wc -l
```
MCP 需要三次调用（read_file → search_text → count_lines），每次都要选工具填参数。

**上下文更省**：训练数据里 CLI 密集，大模型天然会。MCP 的工具 schema 是 2024 年新事物，训练数据少。MCP 需要把工具描述注入上下文窗口，而 CLI "不需要注入、天然就会"是结构性的优势。

### MCP 的两个强项

**不限于文本**：CLI 管道是文本流，对日志、代码、配置文件很高效。但如果处理的是图片、音频、结构化数据库查询结果？文本管道成为瓶颈。MCP 可以传递 JSON 结构、二进制数据、带类型的返回值。

**参数验证**：CLI 里一切都是字符串，拼错了只有执行了才知道，有注入风险。MCP 在调用之前就能校验参数类型，填错了当场报错。

### 隐式 vs 显式

- **CLI**：省显式说明，增加隐式不确定性（当前目录、环境变量、工具版本、shell 类型——Agent 不一定知道）
- **MCP**：要写工具描述、定义参数类型、注入上下文，但执行前提是显式的，输出可预期，出了问题更容易定位

### 两者正在互相靠近

**CLI 这边**：OpenCLI 提供 `cli-manifest.json`（机器可读的命令清单）和 `SKILL.md`（给 Agent 看的能力说明），让 CLI 不再只是"你得先知道有什么"的专家工具。

**MCP 这边**：Code Mode 让 Agent 不一次次只调一个工具，而是直接写一段脚本把多个 MCP 工具串起来。

两者不会合并成同一个——管道组合和结构化调用是两种不同的架构模型，各自在边界上互相学习。

## 三种 CLI 形态

| 形态 | 代表 | 特点 |
|------|------|------|
| **工具型 CLI** | `git`、`docker` | 一直都在，专业世界的连续执行 |
| **平台型 CLI** | 飞书 CLI、OpenCLI | 把网页/平台能力翻译成命令 |
| **Agent 型 CLI** | Claude Code、Codex CLI、Gemini CLI | 表面输入是自然语言，底下调度的是 CLI |

三种形态站的位置相同：**都在执行层**。

## 能力暴露层次

看一个产品时多问一句：**它的能力暴露到了哪一层？**

| 层次 | Agent 能接入吗？ |
|------|----------------|
| 只有 GUI | 人能用，Agent 够不着 |
| 有 API | 能力开放了，但接起来很重 |
| 有 CLI | Agent 可以直接调用，能组合 |
| 有 Skill | CLI + 场景说明，Agent 拿来就能干活 |

层次越完整，Agent 接入的摩擦越小。

# PrfaaS：跨数据中心万亿模型推理架构

> Sources: 云昭 / 51CTO技术栈, 2026-04-26; 月之暗面, 2026-04
> Raw: [prfaas-kimi-cross-datacenter-llm-inference](../../raw/ai-tools/2026-04-26-prfaas-kimi-cross-datacenter-llm-inference.md)

## Overview

月之暗面提出 PrfaaS（Prefill-as-a-Service）架构，证明用普通以太网即可实现万亿参数模型的跨数据中心调度。核心发现是混合注意力模型（Kimi Linear/SWA+GQA）的 KVCache 仅为稠密模型的 1/4~1/36，使跨中心 KVCache 传输成为工程可行。实测：1T 模型吞吐量+54%，P90 首字延迟-64%，跨中心带宽仅用 13 Gbps。

## 问题背景：为什么跨数据中心推理难

传统 PD 分离架构将 Prefill（计算密集）和 Decode（内存带宽密集）分开，但 Prefill 产生的 KVCache 必须通过高带宽网络（如 RDMA/InfiniBand）快速传给 Decode 节点。

**导致的限制**：
- Prefill 和 Decode 必须部署在同一高带宽网络域（单数据中心）
- 异构硬件（H100 Prefill + H20 Decode）难以独立扩展，无法跨低带宽网络高效传输
- 硬件比例固定后难以适应请求长度、缓存命中率的变化

## 关键发现：混合注意力大幅减少 KVCache

新型混合注意力模型（Kimi Linear、SWA + GQA）中，只有少数全注意力层产生随序列长度增长的 KVCache，多数线性复杂度层只产生固定大小状态。

| 模型类型 | KV Throughput（单位时间 KVCache 产出） |
|---------|--------------------------------------|
| 稠密模型 | 1（基准） |
| 混合注意力模型 | 1/4 ~ 1/36 |

**类比**：从"搬运整座山"变成"快递一张光盘"——算法级的"物理瘦身"。

## 架构设计：PrfaaS-PD

### 分层职责

| 集群 | 职责 | 硬件特点 |
|------|------|---------|
| PrfaaS 集群（专用） | 执行计算密集型长上下文 Prefill，通过普通以太网流式传输 KVCache | 高算力、成本优（如 H200） |
| 本地 PD 集群 | 处理短请求和缓存命中的请求，负责 Decode | 带宽优化（如 H20） |

### 混合前缀缓存池（Hybrid Prefix Cache Pool）

混合注意力模型中两类状态性质不同：
- **线性状态**：请求级，大小与输入长度无关，仅完全匹配时复用
- **全注意力 KVCache**：块级，随输入长度线性增长，支持部分前缀匹配

**解决方案**：缓存池将两类状态分开管理，但块大小对齐，允许所有组从共享 KVCache 池分配和释放块。实现跨集群、跨数据中心的 KVCache 高效传输与复用。基于 vLLM 混合 KVCache 管理器扩展。

### 传输介质

PrfaaS 集群与本地 PD 集群通过**普通以太网**（VPC、专线）传输 KVCache，无需 InfiniBand。

## 智能调度

### 选择性卸载

"短请求本地消化，长请求异地处理"：
- 仅当增量长度 > 阈值时才发往 PrfaaS，避免短请求浪费带宽
- 实测阈值约 19.4K token，超过才派发给远端高算力中心

### 带宽感知 + 缓存亲和

- 实时监控 egress 带宽和队列深度，动态调整路由（像导航避开拥堵）
- 若某集群已有部分前缀缓存，优先使用，必要时跨集群传输

### 双时间尺度调度

| 尺度 | 策略 |
|------|------|
| 短时 | 根据带宽和缓存分布动态路由请求 |
| 长时 | 根据流量变化调整 Prefill/Decode 实例比例，重新优化阈值 t |

## 实测数据

**部署**：PrfaaS 集群 32×H200，本地 PD 集群 64×H20，跨集群带宽 100 Gbps 以太网。模型：1 万亿参数级混合架构模型（类 Kimi Linear）。

| 指标 | 结果 |
|------|------|
| 吞吐量 vs 传统部署 | **+54%** |
| 吞吐量 vs 无调度异构 PD | +32% |
| 同等成本吞吐量提升 | ~15% |
| P90 首字延迟（TTFT） | **-64%** |
| 跨中心带宽占用 | 13 Gbps（占 100 Gbps 的 13%） |

**关键结论**：用一根普通 100G 网线，就能在两座城市之间调度顶级算力。

## 行业影响

1. **"东数西算"工程可行**：Prefill 可放在电费便宜的西北，Decode 放在靠近用户的北上广
2. **异构芯片大规模采用**：国产大算力芯片做 Prefill 中心，高带宽芯片做 Decode 中心，PrfaaS 作为"粘合剂"
3. **API 价格下降**：1T 模型处理效率翻倍，千万级上下文成本下降，传导至用户订阅价格
4. **打破"唯显卡论"**：终结推理必须全用 H100 的局面

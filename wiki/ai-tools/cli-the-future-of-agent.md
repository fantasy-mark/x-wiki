# CLI：Agent 时代的执行层

> Sources: 成峰 / AI产品自由, 2026-04-26

## Overview

深入解析 CLI（命令行界面）为何在 Agent 时代重新成为焦点。核心论点：CLI 从一开始就是执行层——解决的是"人/Agent 怎么把动作清楚交给机器"这个根本问题。Agent 时代 CLI/MCP 各有强弱，两者互补而非替代，共同构成 Agent 的执行层基础设施。

## CLI 的本质：从一开始就是执行层

CLI 不是"不够友好但更高效"的界面，而是为解决基础问题而生——**人怎么把动作清楚地表达给机器**。

1970 年代终端出现后，第一次形成"输入一句，立刻看到结果"的实时回路。CLI 随之成形：把动作压成"命令词 + 参数 + 对象"。三个一直没失效的能力：

1. **动作明确交出去** — 把要做的动作压成一行命令
2. **反馈立刻拿回来** — 实时看到执行结果
3. **多个动作继续接起来** — 命令之间可以管道组合

### 三段历史证据

| 证据 | 说明 |
|------|------|
| **rm** | "命令 + 参数 + 对象"的结构从 1970 年代到现在没变过 |
| **Unix** | 给了 CLI 生存哲学：文本进/出、每个命令只做一件事、命令之间可以接起来 |
| **git** | GUI 时代来了以后 CLI 没死，在专业世界里继续承担连续动作和组合动作 |

**核心认知**：CLI 从一开始就是执行层，不是前台界面。GUI 替代的是它的前台位置，不是它的结构优势。

## Agent 时代为什么又轮到 CLI

软件世界从"人操作软件"变成"人提出目标，Agent 拆任务、执行、修正"。

Agent 一旦真正开始干活，立刻碰到四个问题：**命令怎么送进去 → 结果怎么拿回来 → 出错怎么修正 → 多个动作怎么接成链**。CLI 恰好是最成熟的答案——不是因为它最新，而是在这四件事上积累最深。

### CLI 和 API 不是一回事

| | API | CLI |
|--|-----|-----|
| 角色 | 软件能力的原始接口（能力本身） | 能力层上面的包装，封装鉴权/参数拼装/分页/错误处理 |
| 能力层来源 | — | 可以是远程 API（如飞书），也可以是本地程序（如 rm） |

### 为什么执行层先收敛到 CLI

**1. 位置刚好**
- GUI 对人友好，对 Agent 别扭——需要截图/识别界面/定位元素/模拟点击
- 裸 API 太散——鉴权/分页/重试/错误码全得自己处理
- CLI 在中间：比 GUI 稳定（文本接口），比裸 API 省事（封装好了）

**2. 大模型天生会用**
训练语料里充满命令行指令：项目安装说明、构建脚本、技术问答、错误排查。命令调用是在统一的文本格式里做组合；函数调用常常是在一堆结构各异的接口之间切换。

**3. Unix 哲学契合 Agent 工作方式**
- 文本是模型的母语
- stdin/stdout/stderr 通道稳定，exit code 表示成功或失败
- 每个命令只做一件事，多个命令可以管道接成链

```bash
cat access.log | grep ERROR | wc -l
```
"读日志→筛错误→做统计"三件事接成一条链。Agent 擅长的就是这种连续动作——拉数据、填数据、出图表，一步接一步，做一步看一步修一步。

## MCP 和 CLI 的关系

两者都站在 Agent 和底层能力之间，但方式不同：

**CLI 路径**：Agent 必须先知道有什么命令（如 `lark-cli`），生成一行字符串命令，执行后以文本吐回结果。**前提**：Agent 不知道命令存在就走不通。

**MCP 路径**：MCP Server 主动把能力清单推给 Agent，Agent 看到清单选工具填参数发起调用，不需要知道底层跑的是 CLI 还是 API。

### CLI 的两个结构性强项

**管道组合**：一次调用、三步操作通过管道接在一起。MCP 需要三次调用（read_file → search_text → count_lines），每次都要选工具填参数。

**上下文更省**：训练数据里 CLI 密集，大模型天然会。MCP 的工具 schema 是 2024 年新事物，训练数据少。MCP 需要把工具描述注入上下文窗口，CLI "不需要注入、天然就会"是结构性的优势。

### MCP 的两个结构性强项

**不限于文本**：CLI 管道是文本流，处理图片/音频/结构化数据库结果时成为瓶颈。MCP 可以传递 JSON 结构、二进制数据、带类型的返回值。

**参数验证**：CLI 里一切都是字符串，拼错参数只有执行了才知道。MCP 在调用前就能校验参数类型，填错当场报错。

### 隐式 vs 显式

| | CLI | MCP |
|--|-----|-----|
| 优点 | 省显式说明 | 执行前提显式，输出可预期，调试容易 |
| 风险 | 隐式不确定性（目录环境/工具版本/shell类型） | 需要写工具描述/定义参数类型/注入上下文 |

### 两者正在互相靠近

- **CLI 这边**：OpenCLI 提供 `cli-manifest.json`（机器可读命令清单）和 `SKILL.md`（给 Agent 看的能力说明），让 CLI 不再只是"你得先知道有什么"的专家工具
- **MCP 这边**：Code Mode 让 Agent 不一次次只调一个工具，而是写脚本把多个 MCP 工具串起来

两者不会合并——管道组合和结构化调用是两种不同的架构模型，各自在边界上互相学习。

## 三种 CLI 形态

| 形态 | 代表 | 特点 |
|------|------|------|
| **工具型 CLI** | `git`、`docker` | 一直都在，专业世界的连续执行 |
| **平台型 CLI** | 飞书 CLI、OpenCLI | 把网页/平台能力翻译成命令 |
| **Agent 型 CLI** | Claude Code、Codex CLI、Gemini CLI | 表面输入是自然语言，底下调度的是 CLI |

三种形态站的位置相同：**都在执行层**。

## 能力暴露层次

看一个产品时多问：**它的能力暴露到了哪一层？**

| 层次 | Agent 能接入吗？ |
|------|----------------|
| 只有 GUI | 人能用，Agent 够不着 |
| 有 API | 能力开放了，但接起来很重 |
| 有 CLI | Agent 可以直接调用，能组合 |
| 有 Skill | CLI + 场景说明，Agent 拿来就能干活 |

层次越完整，Agent 接入的摩擦越小。**Claude Code 内置能力约 70% 走 CLI 路径，外部服务（GitHub/数据库/第三方平台）约 30% 走 MCP**——底层继续调 CLI，外层用 MCP 接入，这就是 Agent 工具体系最终的样子。
